Decision Table Interpreter

The DecisionTableInterpreter verifies business rules expressed in a rule table by asserting values returned by the system under test against expected values.

Specifying the fixture

The DecisionTableInterpreter follows the general rules for interpreter and fixture selection and fixture naming.

The first row of the table is used to indicate both that we want to use a DecisionTableInterpreter and what fixture to use. The next row is called the header row and serves to distinguish between the types of the columns (this is described in more details in the next section). Finally, the remaining rows, which capture the examples, are the test rows.

Interpretation flow

The rule table is interpreted from top to bottom, one row at a time. Each row is interpreted one cell at a time, from left to right.

Header column

The DecisionTableInterpreter uses the form of the first row - the column header - to decide what to do for each cell. It knows how to distinguish givens from expected values. Given values serve as inputs to the system, whereas expected values serve as comparison values against values that are actually returned by the system.

When a column header ends with special characters ? or (), it denotes an expected value. For all cells within the row, the DecisionTableInterpreter will test the value returned by the system under test against the value specified in the cell, following the rules described in value comparison.

A column header ending with the = character denotes a saved value. In that column, the value returned by the system under development will be saved under the symbol name specified in the cell. When the column header starts with the same = character, it indicates a recalled value. Cells in that column use symbol names to recall previously saved values. Recalled values serve as given values to the system under development.

All other forms of column headers denote a given value. For all cells in the row, the value in the cell will be an input to the system under test using the mechanisms described in action access resolution.

Interpreting given values

Default values

When dealing with given values, the DecisionTableInterpreter uses the default type conversion rules.

In the following example, given values are interpreted by the default type converters that come with LivingDoc:

Blank values

In the case of a blank cell, the value is not converted. It is considered null instead.

cell text input will be?
null

Saving and recalling values

When using saved columns, values returned by the system under development are stored in a variable named after the text of the cell. On execution, the returned value replaces the text of the cell and the cell is ignored.

When using recalled columns, values given by the system under development are recalled from a variable named after the text of the cell.

Cells annotations

When the DecisionTableInterpreter checks a table cell's correctness, it uses standard value comparison and annotates the cell accordingly.

Right and Wrong

When the table cell and the comparison value match, the DecisionTableInterpreter marks the cell "right" using a standard annotation. (See the [annotation specification] for more about standard annotations.)

If the values don't match, the DecisionTableInterpreter uses the wrong annotation and displays the difference between the expected value and the value returned by the system under test.

decision table cell annotation
comparison value returned value annotation?
42 42 right
42 13 wrong

Ignored cells

Empty cells don't trigger any test. They are annotated ignored and display the value returned by the system.

decision table cell annotation
comparison value returned value annotation?
42 ignored
nothing ignored

Unexpected errors

When an unexpected error occurs while comparing the cell value against the actual system value, the DecisionTableInterpreter uses the exception annotation to display information about the error in the cell. Usually a stack trace or similar information is included.

decision table cell annotation
comparison value returned value annotation?
42 error exception
